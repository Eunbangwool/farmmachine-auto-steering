# TOKEN_BUDGET.md — 클로드 토큰 절감 규칙

> 이 파일은 짧게 유지한다. 길어지면 그 자체가 낭비다.
> 적용 대상: ① Claude Code 세션 ② 이 웹 채팅 ③ 코드 작성 스타일.

---

## 0. 한 줄 원칙

**"바뀐 것만, 짧게, 맞는 도구로."** 전체 재업로드·전체 재출력·도구 혼용이 3대 누수.

---

## 1. 작업을 어디서 할지 (가장 큰 절감)

| 작업 | 도구 | 이유 |
|------|------|------|
| 코드 생성·수정·리팩터·빌드점검 | **Claude Code** | repo 직접 연결 → 파일 재업로드 0 |
| 버그 추적·grep·테스트 실행 | **Claude Code** | 로컬에서 검색, 결과만 반환 |
| 전략·기획·문서·하드웨어 분석 | **웹 채팅** | repo 컨텍스트 거의 불필요 |
| 디컴파일·로그 1회성 분석 | **웹 채팅** | 분석 후 결론만 repo에 기록 |

❌ 웹 채팅에 repo 전체 zip 올려서 "점검해줘" = 매번 수천 줄 재독. 1회성일 때만.
✅ 반복 점검은 Claude Code에서 `./gradlew assembleDebug`로 직접 돌린다.

## 2. 무엇을 올릴지 (웹 채팅)

- 전체 파일 ❌ → **해당 함수/블록만** ✅
- "이 에러 고쳐줘" + 전체 로그 ❌ → **에러 줄 + 앞뒤 5줄** ✅
- zip 올릴 땐 **바뀐 파일만** 묶기. 빌드산출물·.git·이미지 제외.
- 긴 로그는 올리기 전에 `findstr`(Windows)로 핵심만 추출.

## 3. 어떻게 요청할지 (출력 토큰)

- "전부 분석/종합 정리" ❌ → "**핵심 3개만**", "**결론만**", "**표로**" ✅
- 코드 요청 시 "**설명 빼고 코드만**" 명시.
- 점검은 "**문제 있는 것만**" — 정상 항목 나열 불필요.
- 여러 질문은 한 메시지에 묶기(왕복 횟수 = 컨텍스트 재독 횟수).

## 4. Claude Code 세션 절감

- `CLAUDE.md`가 길면 매 세션 비용. **요약은 CLAUDE.md, 상세는 별도 .md로 분리**하고 필요할 때만 읽게.
- 한 세션에서 관련 작업을 몰아서. 세션 새로 열 때마다 CLAUDE.md 재독.
- 큰 작업은 한 번에 시키지 말고 단계로 끊되, **각 단계는 한 세션 안에서** 끝내기.
- "전체 빌드 로그 붙여줘" 대신 "**에러 부분만**" 요청.

## 5. 코드 작성 스타일 (간접 절감 — 디버깅 왕복↓)

아래는 "코드가 토큰을 쓴다"가 아니라, **나중에 클로드한테 물어볼 일을 줄이는** 구조.

- **로그를 자체 진단 가능하게**: 상태를 JSON 한 줄로 노출(이미 `statusJson`/`canStatus` 패턴 있음 → 유지).
  → 로그 전체를 클로드한테 올릴 필요 없이 그 한 줄만 보면 됨.
- **실패는 조용히 먹지 말고 사유를 남긴다**: `catch {}` 빈 블록 ❌ → `catch { lastError = ... }` ✅.
  이미 `HardwareInit.lastResult`, `ApolloCanBridge.lastTxOk` 등이 이 패턴. 새 코드도 동일하게.
- **TODO엔 출처/조건 명시**: "// TODO 나중에" ❌ → "// TODO(HW): CAN ID 확정 후 (PROFILE.md §2)" ✅.
  → 다음에 물을 때 맥락 재설명 불필요.
- **매직넘버 금지, 출처 주석**: 이미 VehicleProfile이 모범. 유지.
